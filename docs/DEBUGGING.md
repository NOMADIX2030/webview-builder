# 디버깅·문제해결 가이드

> **목적**: 에러 대응 시 혼란을 방지하고, 근본 원인을 체계적으로 파악하기 위한 **범용 방법론**.  
> **출처**: 2026-02-28 nginx 404 해결 과정에서 추상화한 교훈.

---

## 1. 핵심 원칙 (범용)

### 1.1 원인 분리

증상만 보고 추측하지 않는다. **어디**에서 문제가 발생하는지 먼저 구분한다.

- 문제가 여러 요소(웹서버, DB, API, 앱, 빌드 등)에 걸쳐 있다면, **가장 기본·독립적인 요소**부터 검증
- 예: 웹 404 → 웹서버가 정적 파일을 서빙하는가? / API 에러 → DB 연결은 되는가? / 빌드 실패 → 의존성만 설치하면 되는가?

### 1.2 최소 검증

**가장 단순한 단계**부터 검증한다. 복잡한 설정·코드는 잠시 제외.

- 테스트용 입력·파일·요청으로 "이 단계만" 동작하는지 확인
- 예: 정적 HTML, 단순 API 호출, 최소 설정 파일, 격리된 단위 테스트

### 1.3 단계별 추가

한 단계가 정상 확인된 후에만 다음 단계를 추가한다.

- 1단계 정상 → 2단계 추가 → 2단계 정상 → 3단계 추가 …
- 단계를 건너뛰지 않는다.

### 1.4 한 번에 하나

여러 설정·코드·요소를 **동시에** 수정하지 않는다. 어떤 변경이 효과가 있는지 알 수 없게 된다.

### 1.5 사용자 제안 우선

사용자가 제시한 접근법이 명확하면 **즉시 따른다.** 사용자가 이미 올바른 방향을 알고 있을 수 있다.

---

## 2. 문제 해결 일반 절차

1. **관련 요소 정리** — 프로세스 종료, 설정 비활성화, 의존성 제거 등으로 **깨끗한 상태**에서 시작
2. **최소 단계 검증** — 가장 기본적인 동작만 테스트 (예: 정적 파일, DB 연결, 단순 API)
3. **단계별 추가** — 검증된 단계 위에 필요한 요소를 **하나씩** 추가하며 확인
4. **추가 설정·기능** — 핵심이 정상이면 나머지를 **단계별로** 활성화

---

## 3. 금지 사항 (범용)

| 금지 | 이유 |
|------|------|
| 원인 불명인데 여러 요소를 동시에 수정 | 어떤 변경이 효과가 있는지 파악 불가 |
| 검증 없이 추측으로 설정·코드 변경 | 문제가 악화되거나 꼬임 |
| 사용자 제안을 무시하고 다른 시도 반복 | 올바른 접근법을 놓칠 수 있음 |
| "한 번에 해결" 시도 | 단계를 건너뛰면 원인 파악이 어려워짐 |

## 4. 적용 예시: nginx 404 (2026-02-28)

**증상**: `http://localhost/` → 404 "File not found"

**잘못된 접근** (방법론 미적용):
- 여러 설정을 동시에 변경 (location, root, SCRIPT_FILENAME 등)
- 정적 파일 검증 없이 PHP·앱 설정부터 수정
- 추측 기반 수정 반복 → 설정 꼬임

**올바른 접근** (위 방법론 적용):
1. nginx, PHP-FPM 전부 종료 → 깨끗한 상태
2. 최소 설정만 사용, include 비활성화
3. test-index.html로 정적 파일 200 OK 확인 (1단계)
4. PHP location 추가 → test-index.php 200 OK 확인 (2단계)
5. try_files → index.php 연결 → Laravel 확인 (3단계)
6. include 활성화 → 최종 검증 (4단계)

# AI 협업 프로세스 문서

> **목적**: AI( Cursor 등)와 효과적으로 협업하기 위한 프로세스 가이드  
> **작성일**: 2026년 2월 27일  
> **핵심 원칙**: 계획부터 AI와 함께 논의·분석·문서화한 결과를 기반으로 개발 진행

---

## 1. 개요

### 1.1 이 문서의 역할

| 대상 | 용도 |
|------|------|
| **개발자** | AI와 협업할 때 참조할 프로세스 |
| **AI** | 협업 시 따라야 할 절차 이해 |

### 1.2 핵심 원칙

> **문서는 "AI와 함께 계획한 결과"여야 한다.**  
> AI와 논의·분석·계획을 함께한 뒤 문서로 정리해야, 이후 개발 단계에서 정확히 소통할 수 있다.

---

## 2. 전체 프로세스 개요

```
[Phase 0] 계획·문서화 (AI와 함께)
    ↓
[Phase 1~5] 초기 개발 (문서 기반)
    ↓
[운영 단계] 기능 추가 · 디버깅 · 유지보수
    │
    └── 변경 시 문서 업데이트 → AI와 소통
```

---

## 3. Phase 0: 계획·문서화 (AI와 함께)

### 3.1 목적

AI와 **함께** 요구사항을 논의하고, 기술을 분석하고, 계획을 세워 문서를 만든다.  
이 과정 없이 문서만 있으면 AI가 맥락과 의도를 제대로 이해하지 못한다.

### 3.2 진행 절차

| 순서 | 작업 | AI 역할 | 산출물 |
|------|------|---------|--------|
| 0-1 | **요구사항 논의** | 질문, 대안 제시, 정리 | 요구사항 초안 |
| 0-2 | **기술 스택 분석** | 옵션 비교, 장단점 분석 | 기술 선택 근거 |
| 0-3 | **아키텍처 설계** | 구조 제안, 다이어그램 | 아키텍처 초안 |
| 0-4 | **문서 작성** | 초안 작성, 사용자 피드백 반영 | PROJECT_OVERVIEW.md |
| 0-5 | **구현 명세 작성** | API, DB, 흐름 정의 | DEV_SPEC.md |
| 0-6 | **기술 스택 문서** | 아키텍처 상세화 | TECH_STACK.md |
| 0-7 | **문서 검토·정합** | 일관성 검토, 누락 확인 | 최종 문서 세트 |

### 3.3 AI 소통 예시 (Phase 0)

| 상황 | 예시 지시 |
|------|-----------|
| 요구사항 정리 | "이 서비스 요구사항 정리해줘. 빠진 게 있으면 질문해줘" |
| 기술 선택 | "웹뷰 앱 빌드에 Capacitor vs Cordova 비교해줘" |
| 아키텍처 | "Laravel + Next.js로 이 요구사항 구현하는 구조 제안해줘" |
| 문서 반영 | "방금 논의한 3단계 흐름을 DEV_SPEC에 반영해줘" |

## 4. Phase 1~5: 초기 개발 (문서 기반)

### 4.1 원칙

- **DEV_SPEC.md**의 Phase 1~5 순서를 따른다.
- 단계를 건너뛰지 않는다.
- 각 Phase 완료 후 검증 후 다음 Phase로 진행한다.

### 4.2 Phase별 AI 소통

| Phase | 작업 | 예시 지시 |
|-------|------|-----------|
| **Phase 1** | 프로젝트 초기화 | "@DEV_SPEC.md Phase 1 진행해줘" |
| **Phase 2** | shadcn/ui 설치 | "DEV_SPEC Phase 2대로 shadcn/ui init하고 컴포넌트 추가해줘" |
| **Phase 3** | 백엔드 구축 | "DEV_SPEC 5. API 명세대로 BuildController 구현해줘" |

| 작업 | 내용 |
|------|------|
| **체크리스트 확인** | DEV_SPEC 10.6, 12. 검증 체크리스트 |

| **문서 반영** | 명세와 다르게 구현된 부분이 있으면 문서 수정 |

## 6. 디버깅 및 문제 해결

### 6.0 핵심 원칙 (필수 준수)

> **개발·디버깅·구현 코드 에러 등 문제 발생 시** AI는 반드시 아래를 따른다.

| 금지 | 필수 |
|------|------|
| **추측해서 대응** | 사용자와 **반드시 의논** 후 대응 |
| 표면적 해결 | **근본원인(root cause)** 탐구 |
| 단독 판단 | 공식 문서·GitHub 조사 후 **함께 논의** |

- **추측 금지**: 추측·가정 기반으로 대응하지 않는다.
- **근본원인**: 항상 근본원인을 찾아야 한다.
- **원인 파악이 어려울 때**: 공식 문서, GitHub(이슈, PR, Discussions) 기반으로 조사한 뒤, **사용자와 의논**하여 함께 근본원인을 찾아간다.

### 6.1 원칙

- **에러 메시지·증상**을 명확히 전달한다.
- **관련 문서·코드 위치**를 함께 제공한다.
- **해결 후 문서**에 반영할 내용이 있으면 수정한다.

### 6.2 진행 절차

| 순서 | 작업 | AI 소통 |
|------|------|---------|
| 6-1 | **증상 정리** | 에러 메시지, 재현 방법, 환경 정보 |
| 6-2 | **문서·명세 확인** | "DEV_SPEC 5. API 명세대로 구현했는데 404 나와. 원인 분석해줘" |
| 6-3 | **원인 분석** | AI가 코드·설정·문서 검토 |
| 6-4 | **해결 적용** | 수정 제안 반영 |
| 6-5 | **문서 반영** | 명세 오류·누락이 있으면 문서 수정 |

### 6.3 AI 소통 예시 (디버깅)

| 상황 | 예시 지시 |
|------|-----------|
| **API 에러** | "POST /api/build 호출 시 500 에러. Laravel 로그: [붙여넣기]. DEV_SPEC 5.2 참조" |
| **빌드 실패** | "ProcessBuildJob 실행 시 Capacitor sync 실패. DEV_SPEC 7.2 빌드 단계 확인해줘" |
| **UI 이슈** | "2단계 페이지에서 자동 생성값이 안 표시돼. DEV_SPEC 4.2, 8. 자동 생성 로직 참조" |
| **명세 불일치** | "API 응답 형식이 DEV_SPEC과 달라. DEV_SPEC에 맞게 수정해줘" |

### 6.4 제공할 정보

| 항목 | 내용 |
|------|------|
| **에러 메시지** | 전체 메시지 (가능하면 스택 트레이스) |
| **재현 방법** | 어떤 동작 후 발생하는지 |
| **환경** | OS, Node 버전, 패키지 버전 등 |
| **관련 문서** | @DEV_SPEC.md, @PROJECT_OVERVIEW.md 등 |
| **관련 코드** | 파일 경로, 라인 번호 |

## 7. 유지보수·리팩토링

### 7.1 원칙

- **문서를 기준**으로 "올바른 상태"를 정의한다.
- 리팩토링 후에도 문서와 코드가 일치하는지 확인한다.

### 7.2 진행 절차

| 순서 | 작업 | AI 소통 |
|------|------|---------|
| 7-1 | **목적 정의** | "성능 개선" / "코드 정리" / "구조 재구성" |
| 7-2 | **영향 범위** | 어떤 문서·API·컴포넌트가 바뀌는지 |
| 7-3 | **문서 선반영** | 변경될 부분을 먼저 문서에 반영 |
| 7-4 | **구현** | "문서 변경 내용대로 리팩토링해줘" |
| 7-5 | **검증** | 기존 동작 유지, 문서-코드 일치 확인 |
